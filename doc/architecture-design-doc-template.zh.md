# 架构设计文档模板

## 一句话定义

用一句话说清楚这个系统是什么。

要求：

- 不超过 2 句话
- 不列字段
- 不写实现细节
- 先定义边界

## 目标

这个设计要解决什么问题。

建议写 3 到 7 条。

## 非目标

这个设计明确不解决什么问题。

这一节很重要，因为它防止系统边界不断膨胀。

## 这不是什么

列出最容易混淆的概念，并明确说它们不是这个系统。

例子：

- 这不是长期记忆
- 这不是账号系统
- 这不是运行 Trace
- 这不是后端同步协议

## 作用域和归属

说明数据挂在哪个归属链下面。

例子：

```text
userId
└── workspaceId
    └── windowId
```

要求：

- 区分 scope/id 和 memory layer
- 说明当前实现和目标实现的差异
- 说明哪些 id 是本地 id，哪些 id 是后端 id

## 核心概念

列出系统里的核心概念。

每个概念都要回答：

- 是什么
- 干嘛用
- 是不是用户可见
- 是不是记忆
- 生命周期是什么

## 分类模型

把概念按用途分类。

分类要说明为什么这样分。

不要只列名字。

## 数据模型

如果这个系统会持久化数据，必须写数据库/缓存视角。

至少说明：

- 实体/表有哪些
- 主键是什么
- 外键或归属关系是什么
- 哪些字段是内容
- 哪些字段是元数据
- 哪些字段只是派生索引
- 当前存储和目标存储是否一致

例子：

```text
users
└── workspaces
    └── windows
        ├── window_messages
        └── window_summaries
```

## 生命周期

说明数据什么时候：

- 创建
- 读取
- 更新
- 清理
- 删除
- 归档
- 禁止归档

## 存储设计

分开写：

- 当前实现怎么存
- 目标设计怎么存
- 哪些只本地
- 哪些可选后端同步
- 哪些永远不上传

## 读写规则

说明谁可以读，谁可以写，什么时候可以读写。

要特别写清楚：

- Private mode
- 跨窗口读取
- 长期记忆读取
- 删除确认
- promotion / archive

## 隐私和安全规则

列出安全边界。

例子：

- secrets 不进入长期记忆
- 原始命令输出不进入长期记忆
- Private 内容不上传
- 自然语言不能直接删除长期记忆

## 不变量

写出系统永远必须满足的规则。

这是架构文档里最有价值的部分之一。

例子：

- Private 窗口永不写长期记忆
- Runtime Trace 默认不进入模型上下文
- 当前窗口原始状态不能被其他活动窗口直接读取

## 当前实现和目标实现差异

用表格写清楚：

| 主题 | 当前实现 | 问题 | 目标 |
| --- | --- | --- | --- |

## 决策记录

记录重要选择和为什么这么选。

也要写被放弃的方案。

## 示例场景

写几个端到端例子，说明设计怎么工作。

建议至少包括：

- 普通窗口
- Private 窗口
- 删除
- 归档
- 后端同步
- 用户要求记住
- 用户要求忘记

## 开放问题

还没决定的问题放这里。

开放问题必须具体，不要写成泛泛的 TODO。
